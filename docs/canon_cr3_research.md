# Canon CR3 Support & Research Notes

이 문서는 `Shuttersnap` 프로젝트에서 Canon CR3 RAW 파일 포맷의 메타데이터(특히 셔터 카운트)를 추출하기 위해 진행된 연구 내용과 한계점을 기록합니다.

## 1. 개요 (CR2 vs CR3)
기존 Canon CR2 파일은 표준 **TIFF 기반 구조**를 사용하므로, 파일의 헤더부터 순차적으로 탐색하여 EXIF 및 MakerNote 데이터를 쉽게 추출할 수 있었습니다.
하지만 **CR3 포맷**은 동영상 파일 구조인 **ISO Base Media File Format (ISO BMFF)**를 기반으로 완전히 새롭게 설계되었습니다. 데이터가 여러 개의 독립적인 '박스(Box)' 형태로 쪼개져 저장되며, 이로 인해 메타데이터 파싱의 난이도가 기하급수적으로 상승했습니다.

## 2. 구현된 기능
현재 `extract.ts` 및 `file-uploader.svelte`에 다음과 같은 CR3 지원 기능이 구현되어 있습니다.

### 2.1. `dcraw` 썸네일 에러 방지
* **문제**: `dcraw` WebAssembly 라이브러리는 최신 규격인 CR3 파일의 썸네일 추출을 지원하지 않아, 처리 과정에서 `raw_buf has no thumbnail.` 에러와 함께 프로세스가 중단되는 문제가 발생했습니다.
* **해결**: `.cr3` 확장자를 가진 파일은 `dcraw` 처리를 완전히 건너뛰도록(Bypass) 예외 처리하고, 즉시 기본(Fallback) SVG 아이콘을 표시하도록 하여 앱의 안정성을 확보했습니다.

### 2.2. 기종명(Model) 추출 (`CMT1` 파싱)
* CR3 파일 내에는 여러 개의 메타데이터 박스가 존재합니다. 그 중 첫 번째인 **`CMT1`** 박스는 기존 TIFF의 `Exif IFD0` 역할을 합니다.
* `extract.ts`는 파일의 첫 2MB 내에서 `CMT1` 구조의 TIFF 헤더(`49 49 2a 00` 등)를 스캔하여 **제조사(Make)**와 **기종명(Model)**을 성공적으로 추출합니다.

### 2.3. MakerNote 추출 시도 (`CMT3` 파싱)
* Canon의 고유 메이커노트는 **`CMT3`** 박스 내에 별도의 독립된 TIFF 형식으로 저장됩니다.
* `extract.ts`에 `CMT3` 시그니처(`43 4d 54 33`)를 검색하여 해당 블록을 온전히 분리해 내고, 내부의 IFD 태그를 파싱하는 로직을 성공적으로 구현했습니다.

## 3. 셔터 카운트 추출의 한계점 (미지원 사유)

CR3 파일의 셔터 카운트는 클라이언트(웹 브라우저) 환경에서 완벽하게 추출하기에 치명적인 두 가지 기술적 장벽이 존재합니다.

### 장벽 1: 암호화된 `CameraInfo` 바이너리 블록
최신 Canon 기종(R 시리즈 등)은 셔터 카운트를 표준 텍스트(Int/String) 태그로 저장하지 않습니다.
대신 **`Tag 0x000d` (CameraInfo)** 라는 수천 바이트짜리 알 수 없는 바이너리 덩어리(UNDEFINED, Type 7) 안에 오프셋(Offset) 형태로 숨겨둡니다.
* **EOS R5, R6**: `0x000d` 블록 내부의 **`0x0af1`** 위치에 32-bit 정수로 저장.
* **EOS R50, R8, R6m2**: `0x000d` 블록 내부의 **`0x0d29`** 위치에 32-bit 정수로 저장.
> *이 오프셋 구조를 분석하여 파싱하는 로직은 현재 코드에 반영되어 있습니다.*

### 장벽 2: `CTMD` (Canon Timed Metadata) 트랙의 위치 (가장 큰 문제)
EOS R50과 같은 최신 기종은 위에서 언급한 `CameraInfo (0x000d)` 블록을 파일 앞부분의 `CMT3`에 저장하지 않습니다.
대신, 동영상 구조의 특성을 활용하여 파일 맨 끝자락(예: 30MB 파일의 맨 마지막)에 위치한 **`CTMD` (Canon Timed Metadata)** 라는 동영상 타임라인 트랙 박스 안에 메타데이터를 저장합니다.

* **메모리 최적화의 한계**: `Shuttersnap`은 브라우저 메모리 고갈을 막고 빠른 속도를 보장하기 위해 파일의 **첫 2MB**(`file.slice(0, 2MB)`)만 읽어서 파싱합니다.
* **추출 불가**: 파일 맨 끝에 있는 `CTMD` 트랙을 읽으려면 ① 수십 MB의 RAW 파일을 브라우저 메모리에 통째로 로드하거나, ② 전체 MP4(ISO BMFF) 박스 트리(`moov` > `trak` > `mdia` > `minf` > `stco`)를 순회하는 무거운 비디오 파서를 탑재해야 합니다. 이는 가벼운 웹 도구를 지향하는 본 프로젝트의 스펙을 크게 벗어납니다.

## 4. 결론
* **지원 현황**: CR3 파일은 안정적으로 업로드되며, 기종명과 촬영일자는 정상적으로 추출됩니다.
* **제외 사항**: CR3의 셔터 카운트는 위와 같은 구조적 한계(파일 끝단의 `CTMD` 트랙 은닉 및 `CameraInfo` 바이너리 암호화)로 인해 **추출 불가(미지원)** 상태로 유지합니다. 
* 잘못된 정보(예: `0x0093` FileInfo 태그가 셔터 카운트로 오인되는 현상)가 출력되지 않도록 CR3에서는 해당 태그를 제외 처리하였습니다.
